# 短剧翻译Pipeline架构设计 v2

> 中文短剧出海翻译系统 — 翻译前分析 + 翻译后QA
> 目标语言：英文 (EN)，日文 (JA) 延后
> 日期：2026-03-17（v2 基于 Demo 项目对标后重构）

---

## 1. 系统定位

### 1.1 边界定义

本系统是**翻译前分析工具 + 翻译后质量评估系统**，不涉及配音、口型同步、字幕烧录等后期制作。

```
┌──────────────────────┐    ┌──────────────┐    ┌──────────────────────┐
│   翻译前分析系统       │ → │  人工翻译团队  │ → │   翻译后QA系统        │
│  （本系统 Part 1）     │    │  （外部）      │    │  （本系统 Part 2）    │
│                      │    │              │    │                      │
│  4轮迭代分析          │    │  基于剧本翻译  │    │  信达雅三维评估        │
│  输出：分层剧本包      │    │              │    │  蒸馏专用评估模型      │
└──────────────────────┘    └──────────────┘    └──────────────────────┘
```

### 1.2 输入输出

| 维度 | 描述 |
|------|------|
| **输入** | 中文短剧视频（带现成中文字幕），每集3-6分钟，每部200-500集，每次发布约50集 |
| **Part 1 输出** | 分层剧本包：翻译团队用干净剧本 + 情感提示，机器用的 JSON 元数据分开存放 |
| **Part 2 输入** | 原文剧本（带 ABSA 情感标注）+ 英文翻译结果 |
| **Part 2 输出** | 信达雅三维评分 + 逐句问题标注 + 外包翻译验收报告 |

### 1.3 参考项目

本设计参考了一个已实现的端到端 Drama Translator 项目（ASR → 翻译 → CosyVoice 声音克隆 → Wav2Lip 口型同步 → 字幕擦除），借鉴其核心理念：
- **ASR 锚定多模态 LLM 融合**：FunASR 出精确时间戳，Gemini 看视频出语义信息
- **单一 JSON 中间产物**：`video_description.json` 作为可编辑核心文件
- **轻量脚本编排**：Python 脚本而非重量级 Agent 框架

本系统在此基础上增加：ABSA 多维情感分析、多通道角色交叉验证、翻译后信达雅 QA。

## 2. 总体架构

### 2.1 4轮分析Pipeline

```
Round 1: FunASR
  输入：视频音轨（Demucs 人声分离后）
  输出：时间戳 + ASR文本 + 说话人聚类 (SPK_01, SPK_02...)
  → 写入剧本骨架（时间线 + 对白文本）

Round 2: 多模态 LLM 视频分析（Gemini 3.1 Pro / GPT-5.4 / Kimi K2.5）
  输入：视频 + Round 1 结果
  输出：ASR纠错 + 角色名识别 + 表层情感 + 场景描述
  → 追加 JSON 块到剧本

Round 3: 角色对齐
  输入：Round 1 声纹聚类 + Round 2 角色名 + NER 称谓提取
  输出：character_registry.json（正式称呼 + 别名表 + 声纹映射）
  → 统一剧本中所有 speaker 字段

Round 4: ABSA 情感深度分析
  输入：Round 2 表层情感 + 对白文本 + 音频
  处理：
    第一遍 LLM：内在情绪 + 情绪对象
    第二遍 LLM：语气风格 + 表里对比
    emotion2vec+ 交叉校验强度
  输出：5维 ABSA 情感标注
  → 追加情感 JSON 块到剧本

→ 生成分层输出：
    翻译团队：干净剧本 + 情感提示
    机器存档：完整 JSON 元数据
```

### 2.2 翻译后 QA Pipeline

```
输入：原文剧本（带ABSA标注） + 英文翻译结果

QA 三维评估（信达雅）:
  信：LLM 逐句对比原文和译文语义忠实度
  达：对比 ABSA 情感标注 vs 译文语气，检测情感丢失
  雅：检查用词地道性、文化适配、目标市场习惯

执行方式：
  蒸馏专用评估模型 → 自动评分（也作为外包翻译验收 benchmark）
  LLM 初筛 → 标记低分句
  人工终审 → 只看 LLM 标记的问题行
```

### 2.3 与旧设计的对应关系

| 旧设计阶段 | 新设计位置 | 变化说明 |
|-----------|-----------|---------|
| S1 对白提取 | Round 1 | 保持 FunASR，但定位为时间戳+声纹提供者 |
| S2 角色提取 | Round 2 + Round 3 | Gemini 视觉识别 + 多通道交叉验证 |
| S3 情绪提取 | Round 2 + Round 4 | 升级为 ABSA 5维多轮分析 |
| S4 场景描述 | Round 2 | Gemini 一次性输出，不再单独阶段 |
| S5 剧本生成 | 贯穿全流程 | 剧本即数据，逐轮丰富 |
| S6 情感管理 | Round 4 输出 | 基于 ABSA 数据计算弧线和反转 |
| S7 Hook分析 | Round 2 + 剧本后处理 | Gemini 识别 + LLM 翻译风险评估 |
| Manager Agent | 删除 | 轻量 Python 脚本替代 |
| QA Agent | 翻译后 QA | 从过程QA改为结果QA，信达雅三维 |
| Shared Memory | 文件系统 | character_registry.json + 剧本 Markdown |

## 3. 中间产物：Markdown 剧本 + 嵌入 JSON

### 3.1 设计理念

**剧本即数据库**——一个可读的 Markdown 文档既是人工阅读的剧本，也是机器解析的结构化数据源。多轮迭代逐步丰富，每个 JSON 块标记来源和轮次，通过时间线对齐。

### 3.2 剧本格式示例

```markdown
# EP03 - "步步紧逼"

## 摘要
王刚发现财务报表异常，质问李秘书。李秘书试图掩盖真相，但被王刚识破。

## 角色表
` ` `json
{
  "characters": [
    {"id": "王刚", "title": "农业部部长", "aliases": ["老王","王部长","老公"], "speaker_id": "SPK_02"},
    {"id": "李秘书", "title": "部长秘书", "aliases": ["小李"], "speaker_id": "SPK_01"}
  ],
  "pass": "round3_alignment"
}
` ` `

## 场景一：部长办公室 / 白天
[背景：低沉钢琴曲，窗外阴天]

### 00:01:23 - 00:01:26 王刚
"这份报表上的数字，你能解释一下吗？"
` ` `json
{
  "emotion": {
    "surface": "平静",
    "inner": "愤怒",
    "target": "李秘书的报表造假",
    "tone": "压抑冷笑",
    "intensity": {"surface": 3, "inner": 8}
  },
  "confidence": 0.87,
  "pass": "round4_absa",
  "cross_check": {"emotion2vec": "angry_0.82", "gemini": "suppressed_anger"}
}
` ` `

### 00:01:27 - 00:01:31 李秘书
"部长，这个...可能是录入的时候出了点差错。"
` ` `json
{

    "target": "被发现造假",
    "tone": "结巴掩饰",
    "intensity": {"surface": 6, "inner": 8}
  },

  "pass": "round4_absa",
  "cross_check": {"emotion2vec": "fearful_0.78", "gemini": "nervous_deception"}

` ` `
```

### 3.3 ABSA 情感 5 维度

| 维度 | 说明 | 翻译指导价值 |
|------|------|-------------|
| **表层情绪** | 角色表现出的情绪 | 译文的表面语气 |
| **内在情绪** | 角色真实感受 | 隐含的情绪张力，影响措辞深度 |
| **情绪对象** | 对谁/对什么事 | 帮助翻译团队理解上下文 |
| **语气风格** | 说话方式 | 直接影响英文句式和用词选择 |
| **强度** | 1-10（表层+内在分开） | 量化参考，防止翻译"削弱"情绪 |

### 3.4 分层输出

```
output/batch_01/
  ├── for_translators/              # 翻译团队用（干净可读）
  │   ├── character_guide.md        # 角色表：正式称呼、别名、关系、性格
  │   ├── glossary.md               # 术语表：专有名词建议译法
  │   ├── batch_summary.md          # 批次剧情摘要
  │   ├── translation_notes.md      # 翻译注意事项：文化梗、双关语
  │   └── episodes/
  │       ├── ep001_script.md       # 干净剧本：角色+对白+情感提示（无JSON）
  │       ├── ep001_hooks.md        # Hook分析：类型、翻译风险
  │       └── ...
  │
  └── metadata/                     # 机器用（完整结构化数据）
      ├── character_registry.json   # 角色档案 + 声纹映射 + 别名表
      ├── episodes/
      │   ├── ep001_full.md         # 完整剧本（含嵌入JSON、置信度、交叉验证）
      │   ├── ep001_emotions.json   # ABSA情感完整数据
      │   ├── ep001_timeline.json   # 时间线对齐数据
      │   └── ...
      └── batch_analysis/
          ├── emotion_arcs.json     # 批次情感弧线
          ├── hook_continuity.json  # Hook连贯性
          └── reversal_points.json  # 反转点汇总
```

## 4. 多模态 LLM API 策略

### 4.1 前沿模型现状（2026.03）

| 模型 | 视频理解 | 上下文窗口 | 特点 |
|------|---------|-----------|------|
| **Gemini 3.1 Pro** | Video-MMMU 87.6%，原生视频输入 | 1M tokens | MoE架构，原生多模态，13/16 benchmark 领先 |
| **GPT-5.4** | 原生视频+图像推理 | 1M+ tokens (922K/128K) | 统一推理+多模态+Agent |
| **Kimi K2.5** | VideoMMMU 超 GPT-5.2/Claude Opus 4.5 | 1T MoE | **开源**（Modified MIT），可自部署 |
| **Claude Opus 4.6** | 图像理解强，视频能力有限 | 200K tokens | 推理和文本分析顶级 |

### 4.2 任务分配

| 任务 | 首选 | Fallback | 理由 |
|------|------|----------|------|
| Round 2 视频分析 | Gemini 3.1 Pro | GPT-5.4 / Kimi K2.5 | 需要原生视频输入，Gemini 视频理解最强 |
| Round 3 角色推理 | Gemini 3.1 Pro | GPT-5.4 | 需结合视频画面识别人物 |
| Round 4 ABSA 情感 | Claude Opus 4.6 | GPT-5.4 | 文本深度推理，嘲讽/隐忍等微妙情绪 |
| Hook 分析 | Claude Opus 4.6 | GPT-5.4 | 叙事理解与文化理解 |
| 翻译后 QA | 蒸馏专用模型 | Claude Opus 4.6 | 量化评分需一致性，专用模型更稳定 |

三家 API 互为 fallback，Kimi K2.5 开源可自部署，消除单点依赖。

## 5. 本地微调模型：领域增强

### 5.1 定位

本地微调模型**不是 API 的降级方案，而是领域增强层**。通用 API 模型在短剧垂直领域存在 gap，微调后的专用模型在特定任务上精度超过通用 API。

### 5.2 微调路线

| 任务 | 基座模型 | 微调数据 | 预期提升 |
|------|---------|---------|---------|
| **短剧情感识别** | Emotion-Qwen（Qwen3-VL微调版）/ emotion2vec+ | 1000条人工标注 + 大模型生成扩充 | 嘲讽/隐忍/表里不一等短剧高频情绪，通用模型识别率低 |
| **声纹适配** | CAM++ | 短剧音频（强背景音乐、手机录制） | 低质量音频下声纹聚类准确率 |
| **短剧理解** | Qwen3-VL / InternVL3.5 | Gemini批量标注 + 人工校对 | Hook结构、情感反转、题材套路识别 |
| **信达雅 QA 评估** | 通用 LLM 蒸馏 | LLM评分数据 + 1000条人工标注 | 一致性强、可量化、作为外包验收benchmark |

### 5.3 不需要微调的任务

- **ASR**：FunASR Paraformer 开箱即用 CER 1.95%
- **翻译**：由人工翻译团队完成
- **剧本格式化和摘要**：LLM API 直接够用

### 5.4 微调数据策略

- **冷启动**：1000 条人工标注（情感 ABSA 标注 + QA 信达雅评分）
- **扩充**：大模型（Gemini/GPT/Claude）批量生成，人工抽检校对
- **持续积累**：每次人工翻译+校对的结果自动进入训练池

### 5.5 可微调的开源模型现状（2026.03）

| 模型 | 参数量 | 架构 | 视频能力 | 微调生态 |
|------|--------|------|---------|---------|
| **Qwen3-VL** | ~400B MoE | Hybrid MoE | 最强开源VLM | Emotion-Qwen已有4万+视频情感数据集 |
| **InternVL3.5-78B** | 78B | ViT-MLP-LLM | 接近闭源水平 | 微调友好，成熟工具链 |
| **Kimi K2.5** | 1T MoE | Native multimodal | VideoMMMU SOTA级 | 开源可部署 |
| **emotion2vec+** | ~300M | Self-supervised | 音频情感专用 | 支持下游微调，帧级特征对齐 |
| **CAM++** | 7.2M | ECAPA-TDNN变体 | 说话人声纹 | 3D-Speaker工具链 |

## 6. Round 详细设计

### 6.1 Round 1: FunASR 时间戳 + 声纹

**目标**：获取精确时间戳和说话人聚类，作为后续轮次的锚点。

```
视频文件 → FFmpeg提取音轨 → Demucs人声分离 → FunASR Pipeline
                                                  ↓
                                           ASR文本 + 时间戳(±0.1s)
                                           + 说话人标签 (SPK_01, SPK_02...)
                                           + 标点恢复
```

**模型配置**：
```python
from funasr import AutoModel
model = AutoModel(
    model="paraformer-zh",     # CER 1.95% (AISHELL-1)
    vad_model="fsmn-vad",
    punc_model="ct-punc-c",
    spk_model="cam++"         # 说话人聚类
)
```

**已有字幕处理**：以现有中文字幕为基准文本，ASR 结果用于：
1. 补充说话人标签
2. 发现字幕与实际语音的差异
3. 精修时间戳对齐

### 6.2 Round 2: 多模态 LLM 视频分析

**目标**：Gemini 看视频 + Round 1 结果，一次性输出角色/情感/场景/纠错。

**输入**：视频文件 + Round 1 的 ASR 结果（含时间戳）
**Prompt 策略**：将 ASR 结果作为参考传给 Gemini（ASR 锚定），Gemini 对照画面：
- 纠正 ASR 同音字错误（对照画面字幕）
- 合并被 ASR 拆散的碎片段
- 丢弃噪声误识别
- 识别角色名称（从画面和对白内容）
- 标注表层情感
- 描述场景（环境、灯光、天气）

**输出 JSON 结构**：
```json
{
  "segments": [
    {
      "start": 83.1, "end": 86.3,
      "text_corrected": "这份报表上的数字，你能解释一下吗？",
      "speaker_name": "王刚",
      "surface_emotion": "平静",
      "scene": "办公室，阴天，低沉钢琴曲",
      "timestamp_source": "asr",
      "match_score": 0.92
    }
  ],
  "pass": "round2_gemini"
}
```

### 6.3 Round 3: 角色对齐（多通道交叉验证）

**目标**：将多个来源的角色信息交叉验证，生成统一的角色档案。

```
通道1: Gemini 视觉 → "画面中的丈夫在说话"
通道2: FunASR CAM++ → 声纹聚类 SPK_02
通道3: NER (hanlp) → 对白中出现 "老王", "王部长", "老公"
通道4: LLM 推理 → 基于上下文推断称谓关系

交叉验证:
  Gemini:"丈夫" + 声纹:SPK_02 + NER:"老公"/"王部长"
  → 角色A = {
      正式称呼: "王刚·农业部部长",
      别名: [老王, 王部长, 老公, 亲爱的],
      声纹: SPK_02,
      关系: {李秘书: "上下级"}
    }
```

**跨集维护**：`character_registry.json` 每集处理完增量更新。新别名出现时自动关联或标记人工审核。同一声纹 embedding 在不同集中出现时自动关联。

### 6.4 Round 4: ABSA 情感深度分析

**目标**：为每句对白生成 5 维 ABSA 情感标注。

**多轮 LLM 分析**：
```
第一遍 LLM（Claude Opus 4.6）:
  输入：对白文本 + Round 2 表层情感 + 前后文上下文
  输出：内在情绪 + 情绪对象

第二遍 LLM（Claude Opus 4.6）:
  输入：对白文本 + 第一遍结果
  输出：语气风格 + 表里对比分析

emotion2vec+ 交叉校验:
  输入：对应音频片段
  输出：音频情感类别 + 置信度
  → 与 LLM 结果对比，不一致时标记为 low_confidence
```

**为什么用 ABSA 而非单一标签**：

短剧的情感复杂度远超单一标签。例如：
> "好啊，你真行。"

- 单一标签：`emotion: angry, score: 7` — 丢失了表里对比
- ABSA：`surface: 平静(3), inner: 愤怒(8), target: 对方背叛, tone: 嘲讽` — 翻译团队知道这句需要用冷笑的语气翻译

## 7. 翻译后 QA：信达雅评估

### 7.1 三维评估框架

| 维度 | 定义 | 检测方法 | 示例 |
|------|------|---------|------|
| **信** | 译文忠实原意 | LLM 逐句语义对比 | 漏译/错译/过度意译 |
| **达** | 情感语气还原 | ABSA标注 vs 译文语气对比 | 原文"嘲讽"被译成普通陈述 |
| **雅** | 用词地道自然 | 目标语言母语者视角评估 | 美式 vs 英式，俚语/梗/双关适配 |

### 7.2 蒸馏专用评估模型

**训练路径**：
1. 通用 LLM（Claude/GPT）批量生成信达雅评分数据
2. 1000 条人工标注作为高质量种子
3. 蒸馏到小模型（可部署、低成本、高一致性）

**用途**：
- 日常 QA：自动评分 → 标记低分句 → 人工只审问题行
- 外包验收：作为翻译公司成果的量化 benchmark
- 持续迭代：每次人工校对结果反哺训练数据

### 7.3 QA 执行流程

```
英文翻译结果 + 原文剧本（含ABSA标注）
    ↓
蒸馏专用模型 → 信达雅三维自动评分（逐句）
    ↓
LLM 初筛 → 对低分句生成具体问题描述
    ↓
人工终审 → 只看 LLM 标记的问题行
    ↓
输出：QA报告 + 修改建议 + 验收评分
```

## 附录A: 情感分析详细设计

### 情感评分体系

| 评分 | 强度描述 | 示例 |
|------|----------|------|
| 1-2 | 微弱 | 略有不满、轻微愉悦 |
| 3-4 | 温和 | 平静陈述中带情绪色彩 |
| 5-6 | 明显 | 可被观众明确感知的情绪 |
| 7-8 | 强烈 | 情绪爆发、语调明显变化 |
| 9-10 | 极端 | 暴走、狂喜、崩溃等极端情绪 |

### 情感弧线分析

基于 Reagan et al. 的6种基本故事形状：
1. **Rags to riches** (逆袭上升) — 霸总/逆袭题材常见
2. **Riches to rags** (持续下跌) — 悲剧/虐心集
3. **Man in a hole** (先跌后升) — 单集最常见微结构
4. **Icarus** (先升后跌) — 反转hook经典结构
5. **Cinderella** (升-跌-升) — 跨集常见弧线
6. **Oedipus** (跌-升-跌) — 高能虐心集

### 反转点检测
```
反转 = 内在情感评分在 ≤30秒 内变化幅度 ≥ 4分
       或情感类别从正面跳转到负面（反之亦然）
```

## 附录B: Hook分析详细设计

### Hook类型分类

| 类型 | 描述 | 示例 |
|------|------|------|
| **悬念型** | 关键信息未揭示 | "那个人是谁？" → 下集揭晓 |
| **反转型** | 剧情突变 | 结尾突然出现意外人物/事件 |
| **情感型** | 强烈情绪驱动 | 角色崩溃/告白/背叛 |
| **威胁型** | 危险/冲突升级 | "你有三天时间" |
| **揭秘型** | 部分揭示真相 | 发现关键线索但未完全揭露 |
| **选择型** | 角色面临抉择 | "你选A还是B？" |

### 翻译风险标注
```json
{
  "episode": 15,
  "hook": {
    "type": "wordplay",
    "original": "他说'我会让你好看的'——双关：让你变美/给你好看",
    "translation_risk": "HIGH",
    "risk_reason": "中文双关语，直译丧失悬念效果",
    "suggestion": "需要创译(transcreation)，找等效双关或改用其他hook手法"
  }
}
```
